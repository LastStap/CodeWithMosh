# AGENTS.md — правила роботи Codex

Цей файл описує постійні правила для Codex. Його можна використовувати як глобальний файл `~/.codex/AGENTS.md` або як файл у корені конкретного репозиторію `AGENTS.md`.

## 1. Мова та стиль відповідей

- Завжди відповідай користувачу українською мовою, якщо користувач явно не попросив іншу мову.
- Технічні терміни можна залишати англійською, якщо так точніше: `commit`, `branch`, `DTO`, `repository`, `controller`, `service`, `migration`, `lint`, `test`, `build`.
- Відповіді мають бути практичними: коротко поясни суть, потім дай конкретні команди, файли або diff.
- Не приховуй невпевненість. Якщо чогось не видно з репозиторію або бракує контексту, прямо скажи це.
- Не роби вигляд, що зміни були протестовані, якщо тести фактично не запускались.
- Не пиши довгі теоретичні пояснення без потреби. Для складних змін дай план, ризики та перевірку.
- Для самостійних навчальних завдань не давай майже готовий код, skeleton класу або
  копіпастні фрагменти без прямого прохання користувача. Пояснюй вимоги,
  напрямок розв'язання, обмеження та типові помилки; готовий код давай тільки
  якщо користувач явно просить приклад, рев'ю, fix або допомогу з конкретною
  помилкою.

## 2. Головний принцип роботи

Працюй за схемою:

1. Спочатку зрозумій задачу та швидко оглянь релевантні файли.
2. Перед великими або ризиковими змінами дай короткий план.
3. Внось мінімальні зміни, які вирішують проблему.
4. Після змін запускай релевантні перевірки.
5. Наприкінці дай стислий звіт:
   - що змінено;
   - які файли змінено;
   - які команди запускались;
   - які тести пройшли або чому вони не запускались;
   - що ще варто зробити вручну.

## 3. Дозволені дії без додаткового підтвердження

Codex може виконувати ці дії без окремого підтвердження, якщо вони не змінюють систему поза поточним workspace:

### Читання та аналіз

- Перегляд структури проєкту:
  - `ls`
  - `find`
  - `tree`, якщо встановлений
  - `pwd`
- Пошук по коду:
  - `rg`
  - `grep`
  - `fd`
- Читання файлів:
  - `cat`
  - `sed -n`
  - `head`
  - `tail`
- Аналіз Git-стану:
  - `git status`
  - `git diff`
  - `git log --oneline`
  - `git branch`
  - `git show`

### Безпечні локальні перевірки

- Запуск тестів, лінтерів і білду, якщо вони працюють у межах workspace:
  - `./mvnw test`
  - `mvn test`
  - `./gradlew test`
  - `npm test`
  - `npm run test`
  - `npm run lint`
  - `npm run check`
  - `npm run build`
  - `pnpm test`
  - `pnpm lint`
  - `pnpm build`
- Перевірка версій інструментів:
  - `java -version`
  - `mvn -version`
  - `node -v`
  - `npm -v`
  - `pnpm -v`
  - `git --version`

### Редагування файлів

- Можна редагувати файли в поточному репозиторії, якщо зміна прямо відповідає задачі.
- Можна створювати нові файли документації, тестів, DTO, сервісів, конфігів або міграцій, якщо це логічно потрібно для задачі.
- Не змінюй форматування всього файлу без потреби. Мінімізуй diff.

## 4. Дії тільки після явного підтвердження користувача

Перед виконанням цих дій обов’язково поясни, що саме хочеш зробити, і дочекайся підтвердження:

### Системні команди

- `sudo ...`
- `dnf install ...`
- `dnf remove ...`
- `dnf upgrade ...`
- зміна системних файлів у `/etc`, `/usr`, `/var`, `/boot`, `/opt`
- зміна shell-конфігів:
  - `~/.bashrc`
  - `~/.zshrc`
  - `~/.profile`
  - `~/.config/*`

### Залежності та мережа

- встановлення нових production dependencies;
- оновлення major-версій бібліотек;
- `npm install`
- `pnpm add`
- `mvn versions:*`
- `gradle dependencyUpdates`
- `docker pull`
- `curl ... | sh`
- будь-які команди, які завантажують або виконують код з інтернету.

### Git-операції

- `git commit`
- `git push`
- `git pull`
- `git merge`
- `git rebase`
- `git reset`
- `git reset --hard`
- `git clean`
- `git checkout` з перезаписом локальних змін
- створення або видалення гілок

### Бази даних і Docker

- запуск міграцій проти реальної БД;
- видалення volume/container/image;
- `docker compose down -v`;
- `docker system prune`;
- будь-які команди, які можуть видалити дані.

## 5. Заборонені дії

Codex не має виконувати такі дії:

- Не видаляти файли масово без підтвердження.
- Не запускати `rm -rf` без явного підтвердження.
- Не змінювати `.env`, `.env.local`, `.env.production`, secrets-файли або credentials без підтвердження.
- Не друкувати секрети в чат:
  - API keys;
  - tokens;
  - passwords;
  - private keys;
  - cookies;
  - database URLs з паролями.
- Не пушити код у remote repository без явної команди користувача.
- Не створювати fake-тести, які нічого реально не перевіряють.
- Не вимикати тести, лінтери або security checks, щоб “зеленити” білд.
- Не робити великий refactor, якщо задача просить маленький fix.
- Не міняти архітектуру без пояснення причин і підтвердження.

## 6. Рекомендований режим дозволів Codex

AGENTS.md задає поведінкові правила, але реальні дозволи потрібно виставляти в `~/.codex/config.toml`.

Рекомендований безпечний конфіг для щоденної роботи:

```toml
model = "gpt-5.5"
model_provider = "openai"
model_reasoning_effort = "high"
personality = "pragmatic"

approval_policy = "on-request"
sandbox_mode = "workspace-write"
allow_login_shell = false

web_search = "cached"

[shell_environment_policy]
include_only = ["PATH", "HOME", "USER", "SHELL", "LANG", "LC_ALL", "JAVA_HOME", "MAVEN_OPTS", "NODE_ENV"]

[features]
shell_snapshot = true
multi_agent = true
```

Пояснення:

- `workspace-write` — Codex може читати й редагувати файли у workspace, але не має повного доступу до системи.
- `on-request` — Codex має просити дозвіл для ризикових дій.
- `allow_login_shell = false` — менше ризику, що команди зачеплять глобальні shell-налаштування.
- `web_search = "cached"` — безпечніший режим веб-пошуку за замовчуванням.
- `model_reasoning_effort = "high"` — краще для складного backend/frontend коду, дебагу й архітектури.

Для режиму тільки читання:

```toml
approval_policy = "on-request"
sandbox_mode = "read-only"
allow_login_shell = false
```

Не використовуй `danger-full-access`, `--yolo` або `--dangerously-bypass-approvals-and-sandbox` як стандартний режим.

## 7. Методика роботи з кодом

### Перед змінами

- Перевір `git status`.
- Знайди релевантні файли через `rg`, `fd`, `find`.
- Прочитай існуючу реалізацію перед тим, як створювати нову.
- Якщо є README, AGENTS.md, CONTRIBUTING.md, package.json, pom.xml, build.gradle або docker-compose.yml — враховуй їх.

### Під час змін

- Внось найменший коректний diff.
- Не ламай публічні API без потреби.
- Дотримуйся існуючого стилю проєкту.
- Якщо бачиш декілька способів вирішення, вибирай простіший і надійніший.
- Якщо задача неоднозначна, зроби найрозумніше припущення й явно напиши його у фінальному звіті.

### Після змін

- Запусти найменший релевантний тест.
- Якщо зміна зачіпає багато модулів — запусти повний test/build.
- Якщо тести падають через вже існуючу проблему, покажи конкретну помилку й поясни, чому вона не пов’язана або пов’язана зі змінами.
- Наприкінці покажи, що саме змінилось, але не вставляй величезні diff-и без потреби.

## 8. Java / Spring Boot правила

Для Java, Kotlin і Spring Boot:

- Дотримуйся шарів:
  - Controller — HTTP/API логіка;
  - Service — бізнес-логіка;
  - Repository — доступ до БД;
  - DTO/Mapper — передача даних між шарами;
  - Entity — persistence model.
- Не клади бізнес-логіку в Controller.
- Не повертай Entity напряму з REST API, якщо в проєкті використовуються DTO.
- Для нових endpoint-ів додавай або оновлюй DTO, validation і тести.
- Для PostgreSQL змін у схемі використовуй Liquibase/Flyway, якщо проєкт уже має міграції.
- Не змінюй існуючі міграції, які вже могли бути застосовані. Створюй нову міграцію.
- Для Spring Security не вимика й захист “просто щоб працювало”.
- Для тестів використовуй наявний стиль:
  - JUnit 5;
  - Mockito;
  - Spring Boot Test;
  - Testcontainers або H2, якщо це вже використовується в проєкті.

Рекомендовані перевірки:

```bash
./mvnw test
./mvnw verify
```

або:

```bash
./gradlew test
./gradlew build
```

## 9. TypeScript / Astro / Tailwind правила

Для TypeScript, Astro, React, Node.js і Tailwind:

- Спочатку перевір `package.json`, щоб зрозуміти доступні scripts.
- Якщо є lockfile, використовуй відповідний package manager:
  - `pnpm-lock.yaml` → `pnpm`
  - `package-lock.json` → `npm`
  - `yarn.lock` → `yarn`
- Не змішуй package managers без підтвердження.
- Не додавай UI-бібліотеки без підтвердження.
- Не переписуй весь компонент, якщо можна змінити малу частину.
- Дотримуйся існуючої структури компонентів, стилів і i18n.
- Для Tailwind не створюй зайві кастомні CSS-класи, якщо можна використати utility classes.
- Для Astro не ламай статичну генерацію, routing і локалізацію.

Рекомендовані перевірки:

```bash
npm run lint
npm run check
npm run build
npm test
```

або відповідні `pnpm` / `yarn` команди.

## 10. Fedora / Linux правила

- Для Fedora використовуй `dnf`, а не `apt`.
- Не використовуй `sudo` без підтвердження.
- Не змінюй системні конфіги без підтвердження.
- Не встановлюй глобальні npm-пакети без підтвердження.
- Якщо потрібен системний пакет, спочатку поясни:
  - навіщо він потрібен;
  - яка команда буде виконана;
  - які ризики;
  - як перевірити результат.
- У скриптах використовуй POSIX/Bash-сумісні конструкції, якщо інше не потрібно.
- Не припускай, що користувач працює під root.

## 11. Робота з документацією

- Якщо змінюється поведінка API, CLI або конфігурації — онови README або відповідну документацію.
- Якщо додається нова змінна середовища — додай її в `.env.example`, але не в реальний `.env`.
- Якщо додається новий endpoint — онови OpenAPI/Swagger, якщо проєкт це використовує.
- Документація має бути короткою і практичною.

## 12. Робота з помилками

Коли користувач дає лог або stacktrace:

1. Спочатку знайди головну помилку, а не просто перший рядок.
2. Вкажи найімовірнішу причину.
3. Дай мінімальний fix.
4. Дай команду для перевірки.
5. Якщо є кілька причин — відсортуй їх за ймовірністю.

Не треба одразу переписувати весь код. Спочатку локалізуй проблему.

## 13. Формат фінального звіту

Після виконання задачі відповідай так:

```markdown
Готово.

Змінено:
- `path/to/file` — що саме змінено.
- `path/to/another-file` — що саме змінено.

Перевірка:
- `command` — результат.
- Якщо не запускалось: чому.

Важливо:
- ризики, припущення або наступний крок.
```

Якщо зміни не вносились:

```markdown
Я перевірив код і нічого не змінював.

Знайдено:
- короткий список висновків.

Рекомендую:
- наступні конкретні дії.
```

## 14. Коли оновлювати цей AGENTS.md

Пропонуй оновити цей файл, якщо:

- користувач двічі виправляє тебе з одного й того самого питання;
- у проєкті є повторюваний стиль, який ти не врахував;
- з’явились нові build/test/lint команди;
- змінилась архітектура проєкту;
- з’явились нові правила для Git, CI/CD, Docker або БД.

Не роздувай файл без потреби. Додавай тільки ті правила, які реально допомагають уникати повторних помилок.
